<section class="home-section">
  <h2>How I Use AI</h2>
  <div class="home-grid">
    <article>
      <h3>Claude for the plan</h3>
      <p>Best when the problem is still fuzzy: ideating, pressure-testing assumptions, mapping options, and acting like a trusted advisor before code exists. The risk is stopping at a polished plan instead of turning it into something real.</p>
    </article>
    <article>
      <h3>Codex for the grind</h3>
      <p>Best once there is a direction: implementing, refactoring, writing tests, wiring commands, reviewing diffs, and pushing through the details. The risk is giving it vague work without a tight scope and verification loop.</p>
    </article>
    <article>
      <h3>OpenClaw and local AI for control</h3>
      <p>Best for private exploration, sensitive workflows, offline review, and helping people learn what is possible without handing every task to a cloud service. The risk is using local models where reliability or traceability matters more.</p>
    </article>
  </div>
</section>

<section class="home-section">
  <h2>What This Looks Like</h2>
  <div class="home-grid">
    <article>
      <h3>Legacy platform audit</h3>
      <p>Separate useful infrastructure from inherited complexity, unused licenses, duplicate workflows, and avoidable integration cost.</p>
    </article>
    <article>
      <h3>Purpose-built replacement</h3>
      <p>Design and ship a smaller tool that solves the real job: intake, review, tracking, approvals, reporting, handoffs, or customer operations.</p>
    </article>
    <article>
      <h3>Fast iteration loop</h3>
      <p>Use Claude for planning, Codex for implementation, OpenClaw and local AI for private exploration, and tight feedback cycles to improve the tool while the team is already using it.</p>
    </article>
  </div>
</section>
